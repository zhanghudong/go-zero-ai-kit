# 函数拆分规范

## 适用场景

适用于判断：

- 某段 logic 该不该抽函数
- 某个步骤是否应该保留在主流程
- 某个 helper 文件该不该存在
- 某次拆分是否属于过度设计

## 核心原则

### 1. 先看能不能复用

- 先判断是否可复用。
- 先判断是否属于统一规则。
- 不可复用时，不要为了缩短函数而硬拆。

只有满足下面之一，才值得抽：

- 多个 logic 会用到
- 同一份规则必须统一维护
- 后续明确还会继续复用

### 2. 主函数里保留关键业务步骤

- 主函数中直接保留关键步骤：
- 查了什么
- 解析了什么
- 调了什么服务
- 保存了什么数据

- 不要把查库、解析、调用、保存这类主流程动作为了“封装”再包一层。
- 保证阅读主函数时能直接看出执行顺序。
- 保证报错时能快速定位到主流程步骤。

### 3. 下层已经是函数，上层不要重复包一层

- repo、基础库、公共组件已经提供函数时，logic 层不要再包一层无复用价值的包装函数。
- 仅改变名字、不增加稳定语义、不增加复用价值的包装，视为过度设计。

### 4. 长一点不是问题，重复才是问题

- 不要以“减少行数”作为拆分理由。
- 重点处理下面问题：
- 相同逻辑在多个地方重复
- 同一份字段规则散落在多个地方
- 改一个地方要改很多处

### 5. 统一规则必须抽成公共函数

- 优先抽统一规则，不优先抽一次性动作。
- 只要某段逻辑同时影响下面任一类场景，就统一收口：
- 多个调用入口
- 同一类请求组装
- 同一类校验规则
- 同一类字段映射

### 6. 一次性转换函数，不默认抽

- 当前 logic 内的一次性转换，不默认抽公共函数。
- 只有同时满足“可复用”或“承载统一规则”时才抽。
- 抽出去后如果主流程没有更清楚，就不要抽。

## helper 文件原则

- 默认不创建 `xxx_helper.go`。
- 只有在以下情况下才考虑：
- 同一主题下确实有多组跨文件复用函数
- 这些函数边界稳定
- 文件主题明确，不是泛化的 helper

- 出现下面情况时，说明 helper 职责已经混乱：
- 查库
- 状态映射
- JSON 处理
- 事务保存
- 外部服务调用

## 强制判断标准

### 必须抽

- 多处复用的统一规则
- 明确的领域构建函数
- 多个 logic 共用的固定结构转换

### 不得抽

- 只在当前 logic 出现一次的步骤
- 主流程中的查库、解析、调用、保存步骤
- 只是为了减少行数的一次性包装

## 禁止事项

- 不要为了缩短函数而拆分。
- 不要为了“层次清晰”制造额外跳转。
- 不要新增无主题的 `helper.go`。
- 不要把主流程关键步骤隐藏到无复用价值的包装函数里。

## review 检查项

- 这次抽出来的函数，是否真的能复用。
- 主函数还能不能直接看出查了什么、调了什么、存了什么。
- 这次抽出来的是统一规则，还是一次性动作。
- 抽完之后，阅读路径是不是变长了。
- 如果把函数放回主函数，代码会不会反而更自然。

## 一句话标准

- 先看能不能复用。
- 不能复用时，优先留在主函数里。
- 真正该抽的，是统一规则，不是一次性动作。
