# API 设计规范

适用场景：
- 设计新 `.api`
- 调整接口路径、方法、请求或响应结构
- 评估某个字段是否应该进入 `.api`

核心规则：
- 版本前缀统一使用 `/api/v1`。
- 资源名使用名词，动作语义通过 HTTP Method 表达。
- 查询使用 `GET`，创建使用 `POST`，更新使用 `PUT`，删除使用 `DELETE`。
- 请求/响应结构以 `.api` 为准，由 goctl 生成到 `internal/types`。
- 接口注释应清晰、可用于文档生成。

禁止事项：
- 通过手改 `internal/types` 替代 `.api` 变更。
- 在 `.api` 中写与业务无关的实现细节。
- 路径与方法语义不一致，例如用 `POST` 做纯查询。
