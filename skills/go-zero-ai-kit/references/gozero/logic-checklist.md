# Logic 实现检查清单

严格按照以下清单检查，每条都应明确满足。

适用场景：
- logic 实现完成后做自检
- review 时快速判断是否符合团队标准
- 重构后确认是否出现行为或风格回退

使用方式：
- 本清单用于验收，不负责解释背景。
- 若不确定某条如何落地，回看 `logic-style.md` 和 `logic-patterns.md`。
- 每一项都应能在代码中定位到明确证据。

## 1. 结构
- 保持 goctl scaffold 函数签名与结构不变。
- 使用 `logx.WithContext(ctx)` 初始化 Logger。
- 使用分步注释组织流程，例如 `// ========== 步骤1：... ==========`。

## 2. 输入与校验
- 不做 `req == nil` 判断。
- 不重复做 `.api` 已声明的必填校验。
- 仅在更新类接口中做“字段是否为空”的更新判断。
- 分页参数遵循 `.api` 默认值，不额外判断 `page <= 0`。
- `pageSize` 只做上限裁剪（`MaxPageSize`）。

## 3. 业务流程
- 入口先做必要前置检查，再进入核心流程。
- 核心流程拆成清晰步骤。
- 状态迁移明确写出迁移原因。

## 4. 数据访问
- 查询/写入都处理错误。
- 记录不存在时明确报错。
- 多步骤写入评估事务边界，如需事务需明确说明。
- `CountBuilder` 使用 `*` 作为统计列。
- `squirrel` 条件字段统一加反引号，例如 `` `tenant_id` ``。

## 5. 日志
- 失败场景使用 `Errorw`。
- 关键步骤使用 `Infow`。
- 操作成功类日志使用 `Debugw`。
- 日志字段包含业务主键。
- `logx.Field` 小于等于 3 个时保持单行，超过 3 个再换行。
- DB 查询失败时日志 + 错误包装返回，错误类型按项目规范选择。

## 6. 时间处理
- 使用项目公共时间工具包（如 `timeutil`）处理当前时间获取、解析和格式化。
- 仅在公共工具无法覆盖的 `time.Duration`、layout 常量、超时配置等场景使用标准库 `time`，避免直接写 `time.Now()`。
- 返回给前端的时间字段已完成统一格式化，不直接返回原始 `time.Time`。

## 7. 列表映射
- 使用预分配切片并按索引赋值，避免 `append`。
- 使用统一常量表达布尔或枚举状态，例如 `constant.MockTrue`。
- 常量定义带注释说明。

## 8. ID 与幂等
- 使用 `svcCtx.Snowflake.Generate()` 生成主键。
- 幂等校验通过唯一字段查询。
- 重复写入检测使用项目 `dberr` 工具，例如 `dberr.IsDuplicateEntry`。

## 9. 业务字段补充
- 操作人信息查询用户模型，并区分 `ErrNotFound` 与系统错误；失败必须返回错误。

## 10. 并发与租户
- 状态更新使用条件更新，例如 `WHERE id=? AND status=pending`，并检查影响行数。
- 租户隔离做强校验，不可因请求缺字段而跳过。
