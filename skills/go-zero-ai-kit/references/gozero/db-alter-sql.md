# 数据库新增字段 SQL 规则

当用户提出“给某张表新增字段”的需求时，按本规则执行。

适用场景：
- 用户用自然语言描述某张表要新增一个或多个字段
- 用户希望 Codex 直接生成数据库变更 SQL
- 用户要求把 SQL 写入项目根目录的 `<项目名>.sql` 文件

## 输出文件
- SQL 文件路径固定为项目根目录的 `<项目名>.sql`。
- `<项目名>` 默认取当前项目根目录目录名。
- 若文件不存在，则创建；若文件已存在，则在文件末尾追加新的变更语句。
- 每次新增 SQL 都追加到文件最后，不插入到历史语句中间。
- 在追加 SQL 语句前，先追加一行当天日期注释，作为本次变更分隔。
- 写入前先检查目标文件中是否已存在同表同字段的 `ADD COLUMN` 语句；若已存在，不重复追加。

日期注释示例：

```sql
-- 2026-03-19
```

## SQL 形式
- 使用 `ALTER TABLE ... ADD COLUMN ...` 语句。
- 多个字段默认放在同一个 `ALTER TABLE` 语句中，按字段顺序连续追加多个 `ADD COLUMN`。
- 表名、字段名使用反引号包裹。
- 每条字段定义都必须包含类型、`NOT NULL`、`DEFAULT`、`COMMENT` 和位置子句（`AFTER` 或 `BEFORE`）。
- 字段定义顺序固定为：字段名、类型、`NOT NULL`、`DEFAULT`、`COMMENT`、位置子句。

示例：

```sql
ALTER TABLE `zl_procure_order`
    ADD COLUMN `deferred_status` tinyint(1) NOT NULL DEFAULT 2 COMMENT '暂缓采购状态：1 暂缓 2 不暂缓' AFTER `procure_method`,
    ADD COLUMN `deferred_remark` varchar(200) NOT NULL DEFAULT '' COMMENT '暂缓采购备注' AFTER `deferred_status`;
```

## 字段设计规则
- 先基于字段业务语义生成最合适的英文 snake_case 字段名，不直接照搬中文拼音，除非业务已有强约定。
- 字段名优先短语义；若表名上下文已经足够明确，避免在字段名中重复表级业务前缀。
- 采用短字段名前，先检查当前上下文中的现有列名、历史 SQL 或用户提供信息，避免与已有字段冲突或语义混淆。
- 类型、长度、默认值要优先满足“表达语义明确、兼容已有表风格、实现简单”。
- 若字段未来存在扩展为多状态的可能，优先设计为 `xxx_status`，不要设计成 `is_xxx`。
- 只有在字段语义长期稳定为纯二值开关时，才使用 `is_xxx`。
- 状态/枚举字段优先使用 `tinyint(1)`，默认使用 `1/2` 枚举而不是 `0/1`；其中 `0` 预留给未设置或未参与更新的场景。
- 注释中必须写清楚枚举值含义，例如 `状态：1 暂缓 2 不暂缓`。
- 短文本备注类字段优先使用 `varchar(n)`；若用户给出“200字”这类明确上限，则使用 `varchar(200)`；若用户未给出上限，则默认使用 `varchar(255)`。
- 默认值必须与业务默认语义一致；“默认不是”这类要求要落到 `DEFAULT` 值中。
- 注释必须完整保留业务含义；枚举值说明写入注释。

## 位置规则
- 若用户明确说明“放到某字段后面”，使用 `AFTER`。
- 若用户明确说明“放到某字段前面”，使用 `BEFORE`。
- 若用户未指定位置，默认放到 `create_time` 前面。
- 若默认目标列 `create_time` 在当前上下文中不存在或无法确认存在，则退化为追加到表末尾，不强行生成 `BEFORE \`create_time\``。
- 同一次新增多个字段时：
  - 若用户指定第一个字段的位置，后续字段默认依次放在前一个新增字段后面，保持用户描述顺序。
  - 若用户未指定位置且 `create_time` 可确认存在，第一字段使用 `BEFORE \`create_time\``，后续字段依次 `AFTER` 前一个新增字段。
  - 若用户未指定位置且 `create_time` 不存在或无法确认，第一字段不写位置子句，后续字段依次 `AFTER` 前一个新增字段。

## 执行步骤
1. 从用户输入中提取表名、字段数量、字段中文语义、枚举值、默认值、长度限制和位置要求。
2. 检查项目根目录 `<项目名>.sql` 中是否已存在同表同字段的历史 `ADD COLUMN` 语句，并尽可能检查字段名冲突风险。
3. 为每个字段生成最合适的字段名、类型、长度、默认值和注释。
4. 按位置规则组装单条 `ALTER TABLE` 语句。
5. 写入项目根目录的 `<项目名>.sql` 文件；不存在则创建，存在则在文件末尾先追加当天日期注释，再追加 SQL。
6. 输出时简要说明新增了哪些字段，以及采用这些字段名和类型的原因。

## 约束
- 只生成数据库变更 SQL，不同步修改 model、struct、`.api` 或业务代码，除非用户明确要求。
- 若用户未提供表名，不猜测，先要求补充。
- 若用户给出的字段语义存在歧义，优先选择最常见、最稳定、最容易维护的设计，并在输出中说明假设。
- 若用户指定的位置字段在当前上下文中不存在且无法确认，仍按用户要求生成 SQL，但必须在输出中明确提示用户复核字段顺序。
- 若字段名可长可短，优先选择在当前表语境下不歧义的较短命名，例如在 `zl_procure_order` 中优先 `deferred_status` 而不是 `deferred_procure_status`。
