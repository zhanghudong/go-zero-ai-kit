# 后端架构视角

## 使用场景

适用于判断：

- 服务边界怎么划分
- 数据结构应该放在哪个领域
- 哪类逻辑应该统一收口
- 哪类能力应该抽成公共函数
- 哪类规则必须只有一份实现

## 核心要求

### 1. 先看领域归属

- 函数放在哪个包，不看谁调用，看它属于哪个领域。
- 调用方和归属方不一致时，优先服从归属方。
- 资源域规则属于 `resource`
- 订单聚合逻辑属于 `order`
- 支付处理逻辑属于 `pay`
- 某个领域的标准输入、标准输出、统一规则，放回该领域维护

### 2. 统一规则必须集中

- 只要某份逻辑决定了下面任一类规则，就集中维护：
- 某类请求入参
- 某类校验结构
- 某类状态映射规则
- 某类字段回写规则

- 不要在多个包里各写一份相似实现。

### 3. 主流程优先，抽象其次

- 优先级按下面顺序执行：

1. 主流程清楚
2. 统一规则唯一
3. 领域边界稳定
4. 再考虑是否抽象

- 抽象之后如果主流程更难读，说明抽象做过了。

### 4. 优先返回最终结果

- 公共函数优先返回最终结果。
- 不要让调用方拿半成品后继续手工补字段。

## 常见判断

### 适合放公共函数

- 构建某个领域统一使用的标准输入
- 统一转换领域结构
- 多处依赖的状态映射

### 适合留在主流程

- 当前接口独有的状态判断
- 当前逻辑独有的返回结构组装
- 一次性查询和顺序步骤

## 禁止事项

- 不要把领域规则写到调用方领域中。
- 不要把统一规则分散到多个包。
- 不要为抽象而抽象。
- 不要让公共函数只返回半成品。

## 一句话标准

- 统一规则集中。
- 领域边界稳定。
- 主流程保持可见。
