# 接口开发工作流（严格两阶段）

本流程用于约束 AI 在 go-zero 项目中的接口开发行为，避免在 `.api` 尚未确认、`make` 尚未执行前，提前生成或补全 `handler/logic/types`。

## 阶段一：只修改 `.api`

适用场景：
- 用户要“新增接口”“修改接口定义”“先改 `.api`”
- 项目当前还没有执行 `make` / `goctl api go`
- `handler/logic/types` 还未按最新 `.api` 重新生成

执行规则：
1. 只允许修改 `.api` 文件。
2. 不允许主动创建、修改、补全以下代码：
   - `internal/handler/**`
   - `internal/logic/**`
   - `internal/types/**`
3. 不根据 `.api` 直接手写 scaffold 代码。
4. 完成 `.api` 修改后，明确提示用户先执行 `make`（或项目约定的生成命令）。

阶段一输出：
- 更新后的 `.api` 文件
- 一句明确提示：请先执行 `make`，生成最新 scaffold 后，再继续补 `logic`

## 阶段二：生成后补全代码

进入条件：
- 用户明确表示已经执行过 `make`
- 或仓库中相关 `handler/logic/types` 已按最新 `.api` 生成完成

执行规则：
1. 优先基于 goctl 已生成的 scaffold 补全代码。
2. `handler` 仅做最小必要调整，不在其中编写业务逻辑。
3. 业务实现只放在 `logic` 层。
4. 保持函数签名、目录结构、生成代码风格与项目现状一致。

阶段二输出：
- 补全后的 `logic` 实现
- 如需调整请求/响应结构，只能修改 `.api` 后重新生成；不得手改 `types.go` 等 goctl 生成文件

## 禁止事项

- 在阶段一直接补 `logic`
- 在阶段一抢先写 `handler`
- 在用户未执行生成命令前，假设 scaffold 一定存在且手工补齐整套接口代码
- 手工修改带有 `// Code generated by goctl. DO NOT EDIT.` 标记的文件
- 通过“微调 types”绕过 `.api` 或 goctl 模板的正式生成流程

## 给 AI 的执行口径

如果用户只是让你“开发一个接口”，默认按以下顺序执行：
1. 先修改 `.api`
2. 停止代码继续生成
3. 提醒用户执行 `make`
4. 等用户确认生成完成后，再补 `logic`
